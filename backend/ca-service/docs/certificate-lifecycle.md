# Certificate Lifecycle (`application/CaService.java`)

Explains how certificates are generated, signed, renewed, and expire in the Pulse RMM system.

## Certificate Hierarchy

```
Root CA (self-signed, 10-year validity)
└── Agent Certificate (signed by Root CA, 1-year validity)
    ├── Endpoint 1 (UUID = subject CN)
    ├── Endpoint 2 (UUID = subject CN)
    └── ...
```

The root CA is self-signed and long-lived. Each agent certificate is issued by the root CA, short-lived (1 year), and identified by the agent's endpoint UUID in the certificate's common name (CN).

## Certificate Fields

**Root CA Certificate:**
- Subject CN: `Pulse RMM CA`
- Self-signed (issuer = subject)
- Validity: 10 years from generation (configurable via `CA_ROOT_VALIDITY_DAYS`)
- Key usage: Certificate signing (CA extension)
- Public key: 2048-bit RSA

**Agent Certificate:**
- Subject CN: `{endpointId}` (UUID as string)
- Issued by: Root CA
- Validity: 1 year from issuance (configurable via `CA_AGENT_VALIDITY_DAYS`)
- Key usage: Server authentication, client authentication (mTLS)
- Public key: 2048-bit RSA (generated by agent)

## Enrollment & Initial Issuance

### 1. Agent Generates CSR

On first startup, the agent:
1. Generates a 2048-bit ed25519 keypair locally (never shared)
2. Generates a second 2048-bit RSA keypair (for the certificate)
3. Creates a PKCS#10 Certificate Signing Request (CSR) with:
   - Subject CN = the agent's endpoint UUID (generated locally)
   - Public key = the RSA public key
   - No signature yet (agent signs the CSR to prove it owns the key)

### 2. Agent Submits CSR to CA

Agent makes:
```
POST /api/ca/csr/sign
Content-Type: application/pkcs10

{base64-encoded CSR}
```

Endpoint-service routes this to ca-service.

### 3. CA Service Validates CSR

CaService.signCsr() validates:
- CSR is properly formatted PKCS#10
- CSR signature is valid (agent actually owns the private key)
- CSR subject CN is a valid UUID format
- CSR is not already issued (to prevent duplicates)

### 4. CA Service Signs & Issues

If validation passes:
1. Parses the CSR to extract public key and subject
2. Creates an X509v3 certificate with:
   - Subject CN = the CSR's CN (agent UUID)
   - Issuer CN = "Pulse RMM CA" (root CA)
   - Serial number = random 160-bit value (each cert has unique serial)
   - NotBefore = now
   - NotAfter = now + 1 year
   - Validity = 1 year (from `CA_AGENT_VALIDITY_DAYS`)
3. Signs the certificate with the root CA private key
4. Returns the signed certificate (PEM-encoded) to the agent

### 5. Agent Saves Certificate

Agent receives the certificate and:
1. Validates it (signature, not expired)
2. Saves it to disk: `/var/lib/pulse-agent/agent.crt` (Linux) or `%APPDATA%\Pulse Agent\agent.crt` (Windows)
3. Saves the endpoint UUID separately: `/var/lib/pulse-agent/endpoint.id`

Now the agent can connect to the backend with mTLS using the certificate + private key.

## Certificate Renewal

Agents should request a new certificate before the current one expires (e.g., 30 days before expiry).

### 1. Agent Detects Expiry

Agent checks the certificate expiration on startup and periodically:
- If expiry < 30 days, request renewal
- If expiry < 0 (already expired), error and reconnect with old cert until renewal succeeds

### 2. Agent Requests Renewal

Agent makes:
```
POST /api/ca/csr/renew
Content-Type: application/pkcs10

{base64-encoded new CSR}
```

The CSR is similar to the enrollment CSR, but with the same endpoint UUID.

### 3. CA Service Validates & Issues

CaService.signCsr() validates:
- CSR is valid
- CSR CN matches an existing endpoint (to prevent enrolling new endpoints via renewal)
- Previous certificate for this endpoint is near expiry or expired (prevents over-provisioning)

If valid, issues a new certificate with:
- Same subject CN (endpoint UUID)
- New serial number
- New NotBefore and NotAfter (1 year validity)
- Everything else same as before

### 4. Agent Updates Certificate

Agent receives the new certificate and:
1. Replaces the old certificate on disk
2. Continues using the same private key (same public key)
3. Closes and re-establishes the gRPC stream to the backend with the new cert

## Certificate Expiration

When a certificate expires:
- The agent can still connect, but the agent-hub's TLS validation fails
- Agent-hub rejects the connection with `UNAUTHENTICATED` status
- Agent retries with an older (expired) certificate, fails again
- Agent enters an error loop until manually revoked or certificate is renewed

In practice, the agent should renew before expiry, so this shouldn't happen.

## Certificate Validation Flow

**Agent-Hub on Connection:**
1. Agent connects and presents certificate (mTLS)
2. Agent-hub validates the certificate chain:
   - Root CA certificate is trusted (hardcoded in hub or fetched at startup)
   - Agent certificate is signed by root CA (signature valid)
   - Agent certificate is not expired (NotAfter > now)
   - Agent certificate subject CN is a valid UUID
3. Agent-hub checks the revocation list:
   - Fetch or use cached CRL from ca-service
   - Check if agent certificate serial is in the CRL
   - If revoked, reject with UNAUTHENTICATED
4. Extract agent UUID from certificate CN
5. Accept connection and register in AgentRegistry

## Implementation Details

**CaService (`application/CaService.java`):**
- `signCsr(csrPem)` — Validates CSR, signs with root CA, returns certificate PEM
- `rotateIfNeeded()` — Checks if root CA is near expiry, auto-rotates (future feature)

**CaKeyEncryptor (`infrastructure/CaKeyEncryptor.java`):**
- Encrypts root CA private key with AES-256 before storing in database
- Decrypts on startup or when needed for signing

**CaRootRepository (`infrastructure/CaRootRepository.java`):**
- Spring Data JPA repository for `CaRoot` entity
- Stores one row: root CA public cert (PEM), encrypted private key (base64)

**Dependencies:**
- BouncyCastle — CSR parsing, certificate generation, signing
- Spring Data JPA — Persistence

## Key Operational Questions

**Q: Can I change the root CA?**
A: No, not without significant downtime. To change the root CA:
1. Generate a new root CA with ca-service
2. All agents must re-enroll with the new CA
3. Requires updating all agents simultaneously or running multiple CAs

**Q: What if I lose the CA encryption key?**
A: The root CA private key cannot be recovered. You must:
1. Migrate to a new root CA
2. Re-enroll all agents
3. This is a disaster recovery scenario

**Q: How often should I rotate the root CA?**
A: Every 10 years (default validity period). In practice, you should rotate earlier (e.g., every 5 years) for security. Rotation requires downtime and agent re-enrollment.

**Q: Can agents share a certificate?**
A: No. Each agent gets a unique certificate with CN = endpoint UUID. Sharing would break mTLS identity and revocation.

## Security Considerations

**Certificate Compromise:** If an agent's private key is compromised:
1. Revoke the agent's certificate via ca-service
2. Agent-hub will reject connections from that agent
3. Endpoint remains in the fleet but is unreachable
4. Re-enroll the agent with a new certificate

**Root CA Compromise:** If the root CA private key is compromised:
1. All agent certificates are compromised
2. Revoke all agents immediately
3. Generate a new root CA
4. Re-enroll all agents
5. This is a break-glass scenario; not expected in normal operations
