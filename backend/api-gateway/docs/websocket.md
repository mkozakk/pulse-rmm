# WebSocket Proxies (`config/`, `api/`, `infrastructure/ws/`, `infrastructure/desktop/`)

Bridges browser-based WebSockets to agent gRPC streams for real-time, interactive remote control sessions.

### code
**Configuration**
[`WebSocketConfig.java`](../src/main/java/dev/pulsermm/gateway/config/WebSocketConfig.java):
	- `registerWebSocketHandlers` - Maps specific URL paths (e.g., `/ws/shell`) to their dedicated handlers and interceptors, integrating WebSocket support directly into the Spring Cloud Gateway context.

**Interceptors & Authentication**
[`ShellHandshakeInterceptor.java`](../src/main/java/dev/pulsermm/gateway/api/ShellHandshakeInterceptor.java) & [`DesktopSignalingHandshakeInterceptor.java`](../src/main/java/dev/pulsermm/gateway/api/DesktopSignalingHandshakeInterceptor.java):
	- `beforeHandshake` - Inspect the initial HTTP upgrade request sent by the browser. They extract the temporary session token or JWT, validating it via internal service calls before permitting the connection to upgrade from HTTP to a persistent WebSocket.

**Handlers**
[`ShellWebSocketHandler.java`](../src/main/java/dev/pulsermm/gateway/api/ShellWebSocketHandler.java) & [`DesktopSignalingWebSocketHandler.java`](../src/main/java/dev/pulsermm/gateway/api/DesktopSignalingWebSocketHandler.java):
	- `handleTextMessage` & `afterConnectionEstablished` - Accept the raw, bidirectional data frames sent by the administrator's browser (such as terminal keystrokes or WebRTC SDP offers) and manage the lifecycle of the active browser connection.

**Routers & Registries**
[`ShellSessionRouter.java`](../src/main/java/dev/pulsermm/gateway/infrastructure/ws/ShellSessionRouter.java) & [`DesktopSignalingRouter.java`](../src/main/java/dev/pulsermm/gateway/infrastructure/desktop/DesktopSignalingRouter.java):
	- `routeInput` - Takes text frames received from the browser handler and translates them into Protobuf commands, pushing them down the active agent gRPC stream.
	- `routeOutput` - Receives output events generated by the agent over gRPC and translates them back into text frames to be pushed up the WebSocket to the browser.
[`DesktopSessionRegistry.java`](../src/main/java/dev/pulsermm/gateway/infrastructure/desktop/DesktopSessionRegistry.java):
	- `addSession` & `removeSession` - Maintains an in-memory mapping linking an active browser WebSocket session to the specific target endpoint UUID, ensuring WebRTC signaling data is routed accurately between the two endpoints without broadcasting it to the entire fleet.

### description
Features like the interactive remote terminal and WebRTC desktop control require real-time, bidirectional data flow between the administrator's web browser and the endpoint agent. However, browsers cannot initiate raw gRPC streams, and NAT-bound agents cannot connect directly to browsers. To bridge this gap, the gateway acts as a high-speed protocol proxy, wired together by the `WebSocketConfig`. 

When an administrator requests a terminal session, the browser initiates a WebSocket connection. The `ShellHandshakeInterceptor` first secures this by validating the temporary session token embedded in the upgrade request. Once established, the `ShellWebSocketHandler` assumes responsibility for the browser side of the connection. The architectural complexity is managed by the routing layer (e.g., `ShellSessionRouter`). This router binds the browser's WebSocket session to the agent's gRPC stream located in the Gateway's main agent registry. When the administrator types a command, the handler catches the WebSocket text frame and passes it to the router, which packages it as a Protobuf command and fires it down the gRPC pipe. When the agent replies with terminal output, the Gateway's gRPC observer catches the event, passes it back to the router, which pushes it as a text frame back up the WebSocket to render on the administrator's screen. The `DesktopSessionRegistry` performs a similar function specifically for tracking the complex state of WebRTC peer-to-peer negotiations.
